# openGuiclaw 打包与环境诊断优化总结 (2026-03-05)

## 1. 核心打包逻辑与策略选择
与 `openakita` 将所有底层 Python 环境硬链打包 (`_internal`) 的重量级方案相比，`openGuiclaw` 采取了**“轻启动器 + 运行时化境构建”**的策略：
- **体积优势**：由于不捆绑庞大的 C++ 扩展和几十兆上百兆的深度学习/依赖库，首包体积骤减，更易于分发。
- **环境隔离**：在用户的 `%USERPROFILE%\.openguiclaw\venv` 独立构建运行期环境，不干扰系统全局配置。
- **镜像加速**：借助 `launcher.py` 强制指向高速源（如阿里云镜像 `https://mirrors.aliyun.com/pypi/simple/`），极大地缓解了国内拉包易失败、缓慢的顽疾。

## 2. 异常处理与防死循环设计
在使用轻量化加载的时，最主要的隐患是**弱网或断网环境导致无限重试或彻底卡死**。经过代码核查：
- `launcher.py` 中所有的核心下载环节（Python 安装包下载、`winget` 调用、`pip install` 依赖拉取）均具备严格的**非零错误码阻断机制**。
- 如果安装/下载崩溃（例如 `subprocess.run` 发现错误态），程序会直接全量抛出异常 `RuntimeError`，并通过 `messagebox.showerror` 原生弹窗对用户提示明显的错误信息。
- 用户确认该弹窗后，进程立即通过 `sys.exit(1)` **安全终结退出**，不会陷入后台死循环或无限静默重试。下次依然需要手动排查网络后再次双击重试启动。

## 3. 全新引入的环境诊断体系 (Environment Diagnostics)
尽管规避了无限死循环风险，但如果依赖拉取始终报错（例如使用了错误代理、部分包因本地权限缺失未安装成功等情况），对于黑盒化的 EXE 将非常难排查。因此，项目引入了**环境与依赖健康诊断可视化功能**。

### 3.1 诊断探针 (Backend - `/api/diagnostics`)
由 `core/server.py` 下发底层诊断数据：
- **基础信息**：操作系统、目前调用的真实 Python 路径 (`sys.executable`)、是否处于打包模式 (`frozen`)。
- **连通性探测 (Ping-test)**：通过 `urllib` 毫秒级直接嗅探 `mirrors.aliyun.com`，返回延迟数据和是否畅通的布尔值。
- **代理截获**：打印当前运行时的环境变量：`HTTP_PROXY`/`HTTPS_PROXY`/`ALL_PROXY`，有效预防错误设定的系统代理导致镜像源解析失败。
- **关键库验证**：尝试静默 `import` 测试 `fastapi`,`uvicorn`,`webview`,`playwright`,`mss` 等支撑运行架构核心基础包是否已实装。

### 3.2 可视化反馈与报告生成 (Frontend - 设置界面的系统环境 Tab)
- 融入到前端界面后，在系统层面下增加了“一键诊断环境”卡片。
- 当按下按钮后，快速将后端的探针结果以图形化渲染（绿灯表示一切正常、依赖就位；红灯标示断网或缺包）。
- 同时提供 `/api/diagnostics/export` 端点，用户遇到未知故障可直接将这套结构化结果导出成包含时间戳的文本 `.txt` 发送给开发者。
